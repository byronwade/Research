# Terminology

**Review date:** 2026-07-18
**Status:** canonical vocabulary reference

This reference keeps product, architecture, API, SDK, support, operator, and implementation language consistent. It does not create behavior by itself; governing contracts and requirements define behavior.

## Canonical terms

| Term | Meaning |
|---|---|
| Project | The primary tenant-scoped workspace containing Chat, Documents, Sources, policies, activity, memory, automations, and settings. |
| Chat | The Project control plane where users ask questions, initiate research, review plans, approve work, and inspect progress. |
| Documents | Durable editable Markdown output backed by stable blocks, revisions, citations, collaboration state, and publication projections. |
| Sources | User-authorized evidence inputs such as files, web captures, repository snapshots, connectors, and immutable source versions. |
| SourceVersion | An immutable content-addressed version of a Source with metadata, processing state, rights, retention, and locators. |
| EvidenceSpan | An exact locator inside one SourceVersion that can support, contradict, contextualize, or fail to support a Claim. |
| Claim | A factual assertion with explicit status, evidence relationships, freshness, and publication readiness. |
| DocumentBlock | A stable Markdown block inside a document revision, used for citations, patches, comments, publication, and dependency tracking. |
| DocumentRevision | An immutable snapshot of a document's block membership and serialized Markdown. |
| PublicationSnapshot | An immutable public or private projection of one canonical document revision plus rights, privacy, citation, and support evidence. |
| Research Run | A durable, auditable research operation with scope, plan, source set, model/tool activity, budget, progress, and result resources. |
| ActivityEvent | A content-minimized event describing material Project activity, status, causality, actor, resource, and next action. |
| ActionCard | A review-queue item that asks an authorized user to approve, reject, repair, retry, defer, or inspect blocked work. |
| Command Center | The Project-wide command surface for authorized navigation, search, mode switching, source-scope changes, run control, review actions, automation control, and safe shortcuts. |
| CommandActionDescriptor | A versioned typed description of a command, including target, input schema, result schema, action class, required capabilities, preflight policy, approval class, idempotency scope, and accessibility label. |
| CommandInvocation | A content-minimized record of an attempted command execution, including descriptor version, origin, actor, target, expected version, preflight result, operation or action-card link, status, and idempotency key. |
| Project Action Surface | The server-owned projection of typed Project actions across Command Center, API, SDK, CLI, MCP, native companion, browser extension, connected-app, and Automation Recipe surfaces, generated from canonical descriptors, current policy, preflight, approval, idempotency, expected-version, Activity, and audit rules. |
| ShortcutBinding | A user, Project, Organization, or default key binding that resolves to a CommandActionDescriptor version while preserving conflict detection and authorization policy. |
| Native Companion | The optional desktop shell, browser extension, OS share/import, active-tab capture, selected-context capture, file-watch, notification, deep-link, tray/menu, and global command adapter layer over canonical Project state. |
| Spatial Workbench | The Project-scoped layout surface for task-aware panes, evidence-aware splits, Worksets, progressive hydration, suspend/restore, responsive adaptations, and layout suggestions. |
| Workset | A named, suspendable Project context containing purpose, source scope, selected resource refs, pane layout, snapshots, WorkPacket refs, Focus refs, and restore state without granting resource access. |
| PaneLayoutTemplate | A versioned layout template describing allowed pane types, regions, breakpoints, and accessibility constraints for a work mode. |

| WorksetSnapshot | An immutable restorable point for a Workset containing pane refs, selected resources, source scope, cursors, policy hash, and version refs without storing raw content or granting access. |
| SpatialLayoutSuggestion | A visible, dismissible, reversible recommendation to change Workset or pane layout based on command, WorkPacket, Activity, Trust, source triage, publication, automation, repeated pattern, viewport, or accessibility state. |
| Project Operating Layer | The Project-scoped coordination layer that assembles current work state and next safe actions from authorized Chat, Documents, Sources, Activity, Command Center, Focus, Project Atlas, Trust, Automation Recipes, outcome scorecards, Product Truth, runs, comments, publications, exports, API operations, and support-safe diagnostics without creating a fourth primary surface or second authority. |
| WorkPacket | A rebuildable, viewer-specific projection of current Project work state containing safe refs, versions, status, source scope, current operations, blockers, commands, next-action candidates, redaction summary, cost, latency, approval, and recovery metadata. |

| WorkControlObservation | A content-minimized record of a WorkPacket or NextActionCandidate being shown, opened, dismissed, deferred, invoked, blocked, failed, corrected, accepted, rejected, converted to recipe, or expired. |

- Public-facing copy may use simpler wording, but public docs must still project from the same canonical concept.

## Drift controls

When a term changes:

1. update the governing product or architecture contract first;
2. update this reference;
3. update requirements, API names, SDK names, UI labels, event names, and runbooks if affected;
4. scan for stale synonyms and capitalization drift;
5. run `pnpm docs:check`.

If two terms appear to describe the same durable object, treat that as documentation drift until one term is deprecated or the distinction is defined.
