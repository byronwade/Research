# Requirements traceability matrix

**Review date:** 2026-07-18
**Status:** reference projection, not runtime evidence

This matrix helps humans and agents move from a requirement ID to the governing contracts, owner slice, and validation surface. It does not create new requirements. The canonical requirement statements and owner slices remain in [`../_meta/requirements.json`](../_meta/requirements.json).

Use this reference when:

- a task names a requirement ID;
- a product or architecture document changes a durable obligation;
- an implementation slice needs required reading;
- a launch or support reviewer needs to verify that a requirement has a governing contract;
- a documentation sweep finds a requirement ID that only appears in machine-readable metadata.

## Traceability rules

1. Requirements are stable IDs. Rename prose only when the ID still describes the same durable obligation.
2. Owner slices are primary implementation owners, not the only slices that read the requirement.
3. Product contracts define user-facing behavior. Architecture contracts define how the behavior is built. Delivery and reference docs define validation, release evidence, terminology, schemas, and operational controls.
4. If this matrix conflicts with [`../_meta/requirements.json`](../_meta/requirements.json), update this matrix after correcting the governing contract or metadata.
5. If a requirement changes product behavior, update the governing contract before updating this reference.
