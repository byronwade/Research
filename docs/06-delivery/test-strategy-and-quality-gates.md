# Test strategy and quality gates

Research tests domain invariants, authorization, evidence quality, durable behavior, and user outcomes. A large unit-test count does not compensate for missing tenant, migration, retrieval, or recovery evidence.

`FND-003` makes these gates part of the first implementation slice. The initial monorepo is incomplete until the [`../08-build/foundation-runtime-scaffold.md`](../08-build/foundation-runtime-scaffold.md) gates for type, lint, package boundaries, documentation, generated-file freshness, tests, dependency integrity, secret scanning, supply-chain checks, smoke tests, and provenance run as repeatable commands and CI jobs.

User research, observed-task validation, dogfood limits, perceived-usability measures, accessibility participant coverage, performance-perception evidence, and experience evidence packages are governed by [`user-research-and-experience-validation.md`](user-research-and-experience-validation.md). Human-AI interaction and automation UX reviews for AI capability boundaries, trust calibration, progress, permission, recovery, accessibility, and automation outcome value are governed by [`human-ai-interaction-and-automation-ux-review.md`](human-ai-interaction-and-automation-ux-review.md). Automation failure recovery records, safe next actions, side-effect reconciliation, quiet-wrong outcome handling, and recovery learning artifacts are governed by [`automation-failure-recovery-and-learning-loop.md`](automation-failure-recovery-and-learning-loop.md). Repeatable end-to-end scenario benchmarks that convert those signals into launch evidence are governed by [`research-experience-benchmark-suite.md`](research-experience-benchmark-suite.md). Public, customer, survey, research-literature, official, runtime, generated-summary, and documentation-sweep inputs that affect these gates follow [`public-signal-source-quality-and-citation-policy.md`](public-signal-source-quality-and-citation-policy.md). These records complement automated tests and can block launch even when CI is green.

## Test data

Fixtures are synthetic or explicitly licensed, versioned, deterministic, and isolated by organization. Sensitive production content is not copied into routine test systems. Large-corpus fixtures include PDFs, Office documents, images, audio, video, websites, repositories, spreadsheets, contradictory sources, malicious instructions, and evolving source versions.

## Pull-request gates

A normal slice requires documentation/agent checks, typecheck, lint, unit tests, relevant integration tests, migration checks, dependency review, and changed-path security tests. Affected package boundaries determine additional suites.

## Release gates

A release candidate adds full conformance, migration compatibility, load and capacity, security, accessibility, user-research and experience-validation, restore, rollback, canary, and operational-readiness evidence. Waivers require owner, reason, compensating control, expiry, and follow-up issue. Security, tenant-isolation, and `S0` experience blockers cannot be waived for general availability.

Affected release candidates also include the active `ExperienceBenchmarkScenario` inventory, benchmark runs at the candidate commit, unresolved benchmark blockers, source-quality records for evidence inputs, HumanAIInteractionReview records for changed AI and automation surfaces, AutomationFailureRecoveryRecords and AutomationFailureLearningRecords for changed automation surfaces, OutcomeMetricDefinitions, StrategicBetScorecards, OutcomeReviews for affected strategic claims, ProductTelemetryEventSpecs for telemetry-backed claims, CustomerClaimEvidenceRecords for stronger-than-specification public wording, and regression disposition defined in [`research-experience-benchmark-suite.md`](research-experience-benchmark-suite.md), [`automation-failure-recovery-and-learning-loop.md`](automation-failure-recovery-and-learning-loop.md), [`telemetry-and-experience-instrumentation-matrix.md`](telemetry-and-experience-instrumentation-matrix.md), and [`customer-facing-claim-and-evidence-boundary-matrix.md`](customer-facing-claim-and-evidence-boundary-matrix.md). A scenario benchmark that fails or remains unrun blocks any customer-facing claim that depends on it.

## Flake policy

A flaky test is a defect. Quarantine is time-bounded, owned, visible, and excluded from passing evidence. Retrying until green does not establish correctness.

## Evidence retention

Store test identity, code and fixture versions, environment, artifact digest, configuration, model/provider versions where relevant, results, logs, traces, and approvals. Release evidence must be reproducible and attributable.
