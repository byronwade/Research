# Notebook-style capability benchmark

This benchmark captures the product bar Research must meet when compared with notebook-style research products. It is a planning artifact, not an implementation claim. Product and engineering work must recheck current primary documentation before citing competitor behavior in customer-facing material.

Current public user-opinion and competitor capability signals are tracked in [`user-opinion-and-competitive-signal-audit.md`](user-opinion-and-competitive-signal-audit.md). This benchmark defines the product bar; the signal audit records dated evidence that can change.

## Benchmark purpose

The goal is not to copy another product. The goal is to make sure Research has a clear standard for:

- source onboarding;
- grounded chat;
- citation inspectability;
- generated notes and reports;
- long-form synthesis;
- study and presentation artifacts;
- sharing;
- privacy controls;
- source refresh and maintenance.

## Evaluation dimensions

| Dimension | Research bar |
|---|---|
| Source intake | Users can add authorized files, web captures, repositories, and connected records with explicit capability disclosure. |
| Evidence fidelity | Every factual output resolves to an immutable source version and locator. |
| Chat grounding | Answers expose scope, citations, unsupported claims, and uncertainty. |
| Notes and documents | Outputs become editable Markdown documents rather than trapped chat responses. |
| Long-form synthesis | Large reports use section contracts, evidence ledgers, contradiction checks, and consistency audits. |
| Study artifacts | Audio, maps, tables, timelines, canvases, and other artifacts are typed views backed by the same evidence graph. |
| Sharing | Public and private outputs are projections of canonical content, not forked copies. |
| Maintenance | Source changes revalidate dependent claims and propose minimal patches. |
| Governance | Rights, privacy, residency, provider policy, retention, deletion, and support access remain inspectable. |

## Minimum parity for first production release

Research does not need every artifact type before launch. It does need:

1. simple Project creation;
2. at least one reliable file source path;
3. exact citations from immutable source versions;
4. persistent grounded chat;
5. editable Markdown documents;
6. source and claim status visibility;
7. no public/private drift;
8. export and deletion paths;
9. restore and incident evidence;
10. product truth that matches runtime behavior.

## Differentiation requirements

Research should exceed notebook-style tools in these areas:

- canonical Markdown as a durable interchange format;
- public and private projections from one revision;
- claim-level lifecycle states;
- source-change propagation;
- GitHub repository understanding and patch proposals;
- asynchronous developer API and webhooks;
- explicit provider policy and model routing;
- operator-grade launch evidence.

## Benchmark review process

Before a competitive claim is used:

1. review primary product documentation and support pages;
2. capture the review date;
3. distinguish observed behavior from inferred product direction;
4. record missing evidence and assumptions;
5. keep customer-facing claims conservative.

Benchmark outcomes should feed [`product-readiness-gap-audit.md`](../06-delivery/product-readiness-gap-audit.md), not bypass the implementation plan.
