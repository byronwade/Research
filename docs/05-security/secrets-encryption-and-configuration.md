# Secrets, encryption, and configuration integrity

## Purpose

Research handles connector credentials, model/provider keys, GitHub App material, webhook secrets, database and storage credentials, signing keys, encryption keys, session secrets, support capabilities, and customer-configured integrations. This contract defines how those values and security-sensitive configuration are created, delivered, used, rotated, revoked, audited, and recovered.

## Secret taxonomy

Secrets are classified by authority and blast radius:

- platform root and recovery keys;
- environment deployment credentials;
- service-to-service identities;
- connector and provider credentials;
- tenant/customer-supplied credentials;
- webhook signing secrets;
- session, token, and cookie keys;
- artifact, release, and package-signing identities;
- short-lived agent and support capabilities.

Each type has an owner, storage system, allowed consumers, rotation interval, emergency revocation procedure, backup/escrow policy where applicable, and audit requirements.

## Storage and delivery

Long-lived secrets are stored in an approved secret manager or encrypted provider store, not source control, ordinary database columns, analytics, logs, workflow payloads, support bundles, or client bundles. Runtime delivery uses short-lived identity or scoped environment injection where possible.

Applications receive the minimum secret at the latest practical moment. Parser, browser, code, and repository workers do not receive ambient secrets. Connector workers receive an operation-scoped token through the credential broker, not the stored refresh token.

## Envelope encryption

Highly sensitive customer and connector material uses envelope encryption:

```text
plaintext
→ data-encryption key (DEK)
→ ciphertext + algorithm/version/context
→ DEK wrapped by key-encryption key (KEK)
→ key reference and audit metadata
```

Associated data binds ciphertext to tenant, Project, credential kind, record ID, and schema version so ciphertext cannot be transplanted across contexts. Key identifiers and algorithms are explicit. Cryptographic agility permits new writes under a new key/version while old records remain decryptable during migration.

## Key management

Keys are generated by approved cryptographic systems, never from human-chosen values. Access separates key administration, application use, support, and audit. Key use is logged without exposing plaintext.

Rotation supports:

- new-encrypt/old-decrypt transition;
- bounded asynchronous rewrap or re-encryption;
- progress and failure accounting;
- rollback of metadata while preserving ciphertext;
- emergency revoke and disable;
- validation that old keys are no longer required before destruction.

Root/recovery material has an independently tested restoration procedure and stronger approval than ordinary application keys.

## Application configuration

Configuration is typed, validated at startup and deployment, and separated into:

- non-secret static configuration;
- secret references;
- environment-specific infrastructure identity;
- dynamic operational policy;
- tenant/Project policy;
- feature flags;
- commercial entitlements.

These authorities cannot substitute for one another. Feature flags do not grant authorization, environment variables do not define tenant policy, and billing entitlements do not bypass source rights or security policy.

Unknown, malformed, insecure, or mutually inconsistent production configuration fails closed before serving traffic. Safe defaults are documented and tested. Configuration changes produce an attributable version and diff.

## Environment isolation

Development, preview, staging, and production use separate identities, databases, buckets, signing keys, provider projects, and webhook secrets. Production secrets are not available to preview deployments or pull requests. Test fixtures never contain copied production credentials.

Local development uses `.env.example` with names and descriptions only. Secret-bearing `.env` files are ignored and scanned before commit.

## Connector credentials

Connector credentials belong to the organization/user authorization that created them and are scoped to the provider resources and operations approved. Refresh tokens and private keys are encrypted. Runtime access records tenant, connector, operation, worker, and expiration.

Revocation immediately blocks new token issuance, cancels eligible queued work, marks affected Sources or repositories stale/unavailable, and preserves historical evidence according to authorization policy. Reauthorization creates a new credential version; it does not silently reactivate old grants.

## Webhooks and signing

Webhook secrets are versioned and support overlap during rotation. Verification checks algorithm, signature, timestamp window, delivery ID, replay state, expected issuer, and body bytes before parsing. Signing key changes are propagated without accepting unsigned fallback.

Public API keys and service accounts are hashed or encrypted as appropriate, displayed once, scoped, expiring where practical, revocable, and attributable to usage.

## Logging, telemetry, and support

Known secret patterns and configuration values are redacted before logs, traces, errors, analytics, AI prompts, or support bundles leave the process. Redaction is defense in depth; secret-bearing payloads should not enter telemetry APIs at all.

Debug modes that expose headers, environment, SQL bind values, model prompts, tool arguments, or provider payloads are disabled in production unless a separately authorized, time-bounded diagnostic path performs content minimization.

## Rotation and incident handling

Rotation is automated and rehearsed for high-value credentials. A rotation event records old/new versions, scope, dependent services, validation, and completion. Compromise response can identify all deployments, runs, connectors, webhooks, and external effects associated with a credential version.

Emergency rotation must not require rebuilding unrelated application artifacts. After rotation, the system validates provider access, workflow resumption, webhook verification, publication, and support/admin operations.

## Definition of done

Secrets and configuration are production-ready when no ordinary developer or runtime process needs broad ambient credentials; secret versions can be located, rotated, revoked, and audited; ciphertext is context-bound; environments are isolated; startup rejects insecure configuration; and compromise response can contain the exact blast radius without customer-content inspection.
